# 会聊天 — 开发规范（Code Spec）

> 融合「软件开发团队」多角色协作 + 「SuperPowers」TDD/调试/分支管理 + 「Karpathy 编程指南」行为约束 + 「Matt Pocock」ADR/共享语言/架构审查。
> 本文件是 PLAN_v4.md 的工程执行层，定义 **怎么写代码**，而非 **做什么功能**。

---

## 一、工作流概览

```
需求确认 → 架构设计 → 任务分解 → TDD实现 → 代码审查 → QA验证 → 分支合并
   ↑                                                              ↓
   └──────────────── 发现设计/需求问题 ← 反馈回路 ←─────────────────┘
```

## 二、阶段定义 + 约束

### Phase 1：需求确认（产品经理 — 许清楚）

**触发条件**：新功能、功能变更、增量需求

**产出**：PRD 文档
- 简单 PRD（默认）：产品目标 + 用户故事 + P0/P1/P2 需求池 + UI 设计稿 + 待确认问题
- 完整 PRD（竞品/市场分析时）：额外包含竞品分析 + 市场定位

**门控**：PRD 中的「待确认问题」全部确认后，方可进入架构设计。

---

### Phase 2：架构设计（架构师 — 高见远）

**触发条件**：PRD 已确认

**产出**：系统设计文档，一次性输出以下全部：
- 实现方案 + 框架选型
- 文件列表及相对路径
- 数据结构和接口（类图）
- 程序调用流程（时序图）
- **任务列表**（有序、含依赖关系、按实现顺序排列）
- 依赖包列表
- 共享知识（跨文件约定）
- 待明确事项

**门控**：「待明确事项」全部解决后，方可进入任务分解。

---

### Phase 3：TDD 实现（工程师 — 寇豆码）

> **本节采用 SuperPowers 的 TDD 铁律 + Karpathy 的目标驱动执行。**

#### 3.1 TDD 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

写在测试之前的代码 = 删除，从零开始。不保留为"参考"，不"边写边改"。

#### 3.2 目标驱动转化（采用 Karpathy 原则4）

**把命令式任务转化为可验证目标**。不要告诉代码"做什么"，给它"什么算成功"：

| 命令式（弱） | 可验证目标（强） | 验证点 |
|------------|---------------|--------|
| "加输入验证" | 写无效输入的测试，让它们全部通过 | `pytest tests/test_validation.py -v` → ALL PASS |
| "修这个 Bug" | 写复现 Bug 的测试，让它从红变绿 | `pytest -k "test_regression"` → RED→GREEN |
| "重构 X 模块" | 重构前后全量测试保持绿色 | `pytest tests/x/` → PASS 不变 |

**多步骤任务格式**：每步声明验证点
```
1. [步骤描述] → verify: [pytest 命令 + 预期结果]
2. [步骤描述] → verify: [pytest 命令 + 预期结果]
3. [步骤描述] → verify: [pytest 命令 + 预期结果]
```

**强成功标准**让 AI 自主循环。弱标准（"让它能工作"）需要不断澄清。

#### 3.3 红-绿-重构循环

**每个任务**执行以下 5 步：

| 步骤 | 动作 | 验证 |
|------|------|------|
| 🔴 RED | 写一个最小失败的测试 | `pytest` 确认 FAIL |
| ✅ VERIFY RED | 确认失败原因正确（功能缺失，非语法错误） | 预期失败信息 |
| 🟢 GREEN | 写最简代码让测试通过 | 不写额外功能 |
| ✅ VERIFY GREEN | 运行全量测试 | 目标测试 PASS + 旧测试全部 PASS |
| 🔵 REFACTOR | 去重、改名、提公共函数 | 全程保持测试绿色 |

#### 3.4 测试质量标准

| 要求 | 好 | 坏 |
|------|-----|-----|
| 最小化 | 一次测一个行为 | 名称里有"and" |
| 清晰 | 描述行为 `test_retries_failed_operation` | `test1`、`test_works` |
| 真代码 | 不 mock 除非不可避免 | 全 mock 导致测的是 mock |

#### 3.5 全局一致性审查

全部文件完成后，工程师执行全局一致性审查（IS_PASS: YES/NO）：
- IS_PASS: NO → 修复问题后重新审查（最多 2 轮）
- IS_PASS: YES → 生成代码摘要，交给 QA

---

### Phase 4：QA 验证（QA 工程师 — 严过关）

**触发条件**：代码通过全局一致性审查

**产出**：
- 为核心模块编写测试用例
- 运行测试并做出 **智能路由判定**：

| 判定结果 | 行动 |
|---------|------|
| 源码有 Bug | 反馈给工程师（寇豆码）修复 |
| 测试代码有 Bug | QA 自行修复 |
| 全部通过 | 报告成功 |

**最多 2 轮测试**：第 1 轮发现问题反馈修复，第 2 轮回归验证。2 轮仍不过则输出报告标注遗留问题。

---

### Phase 5：分支完成（采用 SuperPowers 分支流程）

**触发条件**：所有任务完成，所有测试通过

**流程**：
1. 确认 `pytest` 全量通过
2. 确定 base 分支
3. 四选一操作：
   - A. 本地 merge
   - B. push + PR
   - C. 保留分支（暂不合入）
   - D. 丢弃分支
4. 执行选择的操作
5. 清理临时文件

## 三、代码行为约束（采用 Karpathy 编程指南）

> 以下三条原则来自 Andrej Karpathy 对 LLM 编程常见陷阱的总结，约束的是 **写代码时的思考和行为方式**，而非流程步骤。

### 原则1：先思考再编码（Think Before Coding）

**不要假设。不要隐藏困惑。呈现权衡。**

编码前必须：
- **陈述假设**。如果对任何事不确定，停下来问。不要沉默猜测。
- **呈现多种解释**。如果一个问题有多种理解方式，列出来——不要无声选择其中一种。
- **指出更简单方案**。如果存在比当前方案更简单的实现方式，说出来。该 push back 就 push back。
- **困惑即停**。如果某处不理解，停下来。说出困惑点。问清楚再动手。

反模式示例：
```
❌ 用户说"加个登出按钮"，AI 默默决定用 OAuth 2.0 + JWT 方案，写了 300 行
✅ "登出功能有几种实现方式：1）清除本地 session（简单，10行）2）服务端 token 失效 + 单点登出（完整，150行）。你倾向哪种？"
```

### 原则2：简洁优先（Simplicity First）

**用最少代码解决问题。不写任何推测性内容。**

- 不写未被请求的功能。
- 不为仅使用一次的代码创建抽象（类、接口、工厂）。
- 不写用户没要求的"灵活性"或"可配置性"。
- 不处理不可能发生的错误场景。
- **如果你写了 200 行而它可以精简到 50 行，重写它。**

自检问题："一个高级工程师会觉得这段代码过度复杂吗？" → 如果是，简化。

反模式示例：
```
❌ 一个简单的配置读取 → 创建 ConfigManager 单例类 + ConfigInterface 抽象 + 工厂模式
✅ 一个简单的配置读取 → def load_config(path): return json.load(open(path))
```

### 原则3：精准修改（Surgical Changes）

**只触碰必须改的。只清理自己造成的烂摊子。**

编辑已有代码时：
- ❌ 不要"顺手改进"相邻的代码、注释、或格式。
- ❌ 不要重构没坏的东西。
- ❌ 不要删除你发现有问题的预存死代码——提出来，但不动手。
- ✅ 匹配已有的代码风格，即使你觉得你的方式更好。
- ✅ 只清理你自己造成的孤儿引用（你的改动导致不再使用的 import / 变量 / 函数）。

**检验标准**：diff 中的每一行改动，都必须能直接追溯到用户的请求。如果某行改动解释不了"为什么改这个"——那就不该改。

反模式示例：
```
用户要修一个 "忘记密码" 按钮的颜色 Bug

❌ diff 里出现：改了按钮颜色 + 重构了整个 auth 模块 + "优化"了 3 个相邻组件 + 换了 import 顺序
✅ diff 里出现：只改了一个 CSS 变量的值
```

## 四、系统调试流程（SuperPowers 四步法 + Matt Pocock 困难 Bug 升级）

> **约束**：发现 Bug 后，在定位根因之前，**不得写任何修复代码**。

### 标准流程（适用普通 Bug）

| 阶段 | 动作 | 产出 |
|------|------|------|
| **1. 根因调查** | 读错误信息、复现 Bug、检查最近变更、追踪数据流 | 确认根本原因（非症状） |
| **2. 模式分析** | 找项目中类似的正确实现、对比差异 | 差异分析报告 |
| **3. 假设 + 验证** | 一次一个假设，用测试证实/证伪 | 验证结论 |
| **4. 修复 + 回归** | 在根因层面修复，运行全量测试 | 修复确认 + 无副作用 |

### 困难 Bug 升级路径（采用 Matt Pocock /diagnose 六阶段）

> **触发条件**：标准流程走完未定位根因，或 Bug 难以稳定复现、涉及性能回归、跨模块耦合。

| 阶段 | 动作 | 关键约束 |
|------|------|----------|
| **1. 构建反馈回路** | 构造一个快速、确定性、AI 可运行的通过/失败信号 | **这是核心**。2 秒确定性回路 = 调试超能力。30 秒 flaky 回路 ≈ 没有回路 |
| **2. 复现** | 运行回路，确认看到用户描述的故障模式 | 确认复现的是同一个 Bug，不是相似的另一个 |
| **3. 假设** | 生成 3-5 个排名假设，每个必须可证伪 | 格式："如果 X 是根因，那么改 Y 会让 Bug 消失/改 Z 会让它更糟" |
| **4. 仪器化** | 一次只改变一个变量，用断点/标记日志区分假设 | 所有调试日志加唯一前缀 `[DEBUG-xxxx]`，事后一次 grep 清理 |
| **5. 修复 + 回归测试** | 先写回归测试再修代码 | 如果没有合适的测试缝 → 这本身就是发现：架构阻碍了 Bug 锁定 |
| **6. 清理 + 事后分析** | 删除仪器代码、确认原场景不再复现、记录根因 | 问：如何防止此类 Bug 再次发生？→ 如有架构改进建议，提交定期架构审查 |

**反馈回路构建技巧**（优先顺序）：
1. 失败测试（任何缝）→ 2. curl/HTTP 脚本 → 3. CLI + fixture → 4. 无头浏览器 → 5. 重放 trace → 6. 最小 harness → 7. 属性/模糊测试 → 8. 二分查找 → 9. 差分对比 → 10. HITL 人工脚本

## 六、文件与文档规范

| 文档类型 | 路径格式 | 用途 | 适用阶段 |
|---------|---------|------|---------|
| 产品规划 | `PLAN_v4.md`（主文档，持续迭代） | 产品层面规划 | 全阶段 |

- PLAN_v4.md 仅存放产品层面的规划，技术和工程文档独立成文件。

- ADR 仅在同时满足三个条件时创建：①难逆转 ②无上下文会令人惊讶 ③经过真实权衡。

## 八、禁止事项

- ❌ 测试没红就写代码（TDD 铁律）
- ❌ Bug 不定位根因就改代码（系统调试四步法）
- ❌ 跳 commit hooks
- ❌ 写代码同时写测试（必须先测试后代码）
- ❌ QA 发现源码 Bug 后工程师不修复就继续
- ❌ 跳过全局一致性审查直接交付
- ❌ 不确定时沉默猜测、不陈述假设（Karpathy 原则1）
- ❌ 过度设计——200 行能解决的问题写 500 行（Karpathy 原则2）
- ❌ 顺手"改进"相邻代码、重构没坏的东西（Karpathy 原则3）
- ❌ 多步骤任务不声明验证点（Karpathy 原则4）
- ❌ 术语不精确时沉默接受、不查 CONTEXT.md（Matt Pocock）
- ❌ 不可逆的架构决策不留 ADR 记录（Matt Pocock）

## 九、架构决策与共享语言（采用 Matt Pocock 机制）

> **使用阶段**：规划阶段 + 架构设计阶段。为整个项目建立共享领域语言和可追溯的决策链。

### 9.1 CONTEXT.md — 项目领域词汇表

位置：项目根目录 `CONTEXT.md`

**定位**：纯词汇表，不含实现细节。不是规格说明、不是草稿、不是 todo list。

**何时创建**：第一个术语被明确时惰性创建。不要一上来就写空模板。

**何时更新**：
- 产品经理（许清楚）的 PRD 中出现新概念时，提炼为规范术语
- 架构师（高见远）的架构设计中明确模块边界时
- 任何阶段发现模糊或过载的术语时，立即精确化

**示例**：
```
之前："发一个消息给对象"
之后："生成 Chat 回复"（"Chat" 是 CONTEXT.md 中定义的标准术语）
```

**价值**：统一命名 → 变量/函数/文件名一致 → 代码库易导航 → AI 代理少消耗 token 在歧义澄清上。

### 9.2 ADR — 架构决策记录

位置：`docs/adr/<序号>-<决策简称>.md`

**触发条件**（三个全满足才创建）：
1. **难逆转** — 改主意的成本很高
2. **无上下文会令人惊讶** — 未来的读者会问"为什么这样做？"
3. **经过真实权衡** — 确实有替代方案，你为了具体原因选了其中一个

**格式**：
```markdown
# ADR 0001: [决策标题]

**状态**：已采纳

**背景**：[为什么需要做这个决策]

**决策**：[做了什么选择]

**备选方案**：[考虑过但没选的方案 + 不选的理由]

**后果**：[正面和负面的后续影响]
```

**示例触发场景**：
- ✅ Phase 0 选 JSON 文件而非 SQLite → 影响后续迁移，有真实权衡 → 创建 ADR
- ✅ 选择 Vite+React 而非 Next.js → 影响渲染策略和部署 → 创建 ADR
- ❌ 变量命名风格 → 易逆转 → 不创建 ADR
- ❌ 显而易见的决策（如"用 HTTPS"） → 不创建 ADR

### 9.3 与 DEV_SPEC 现有流程的衔接

```
Phase 1（需求确认）
  └── 产品经理 PRD → 新概念提炼进 CONTEXT.md

Phase 2（架构设计）
  └── 架构师输出 → 不可逆/反直觉决策 → 创建 ADR
  └── Grill 质询（可选）→ 用 CONTEXT.md 的术语挑战设计一致性

Phase 3-5（开发）
  └── 遇到架构摩擦 → 记录，不立即重构（Karpathy 原则3）
  └── 定期 Architecture Review → 读取 ADR 和 CONTEXT.md 作为输入
```

## 十、开发阶段补充实践（采用 Matt Pocock 机制）

> **使用阶段**：开发阶段。这些实践在代码编写过程中启用，不在规划阶段执行。

### 10.1 定期架构审查（/improve-codebase-architecture）

**触发**：每完成一个 Phase 或每积累 10+ 个文件后，进行一次架构审查。

**目标**：将浅层模块深化为深层模块——让少量接口驱动大量行为。

**审查标准**（使用 Matt Pocock 的深度模型语言）：
- **模块（Module）** — 有接口+实现的任何东西
- **深度（Depth）** — 大量行为藏在少量接口背后
- **缝（Seam）** — 行为可被改变而不修改原代码的位置
- **局部性（Locality）** — 变更、Bug、知识集中在一个地方

**执行**：
1. 遍历代码库，寻找摩擦点：
   - 理解一个概念需要跳多个小模块？
   - 模块是否"浅"——接口几乎和实现一样复杂？
   - 紧密耦合的模块是否跨缝泄漏？
   - 哪些部分没测试或难以通过当前接口测试？
2. 对可疑模块做**删除测试**：想象删除它——复杂度消失（过路模块）还是复发到 N 个调用者（值回票价）？
3. 输出候选深化列表，标注推荐强度（Strong / Worth exploring / Speculative）
4. 用户选择一个候选后，进入 grilling 对话走设计树

**频率建议**：Phase 0 先不做（代码量少），Phase 1 及之后每隔几天运行一次。

### 10.2 全局视角分析（/zoom-out）

**触发**：面对不熟悉的代码段需要理解其在整个系统中的位置时。

**行为**：跳出局部，提供更广泛的上下文——这个模块在整个系统中承担什么角色、和哪些模块协作、为什么存在。

### 10.3 Git 安全保护（/git-guardrails）
